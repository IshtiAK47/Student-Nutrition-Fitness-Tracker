PROJECT TITLE
================================================================================
Student Nutrition and Fitness Tracker - University Canteen Management System


1. PROJECT OVERVIEW
================================================================================

Purpose:
A comprehensive Database Management System (DBMS) developed for tracking 
student nutrition intake and fitness records in a university canteen 
environment. The system manages food purchases, calculates nutritional values, 
monitors health metrics, and generates analysis reports.

Academic Context:
- Course: Database Management Systems (4th Semester BSc in ICT)
- Focus: Database design, normalization, SQL queries, data analysis
- Evaluation Criteria: Database structure (3NF normalization), complex SQL 
  queries, transactions, data integrity

Target Users:
1. Canteen Staff: Create invoices, manage food purchases
2. Medical Officers: Monitor student health, analyze nutrition
3. Teachers: View student nutrition reports, health records

3. DATABASE DESIGN
================================================================================

Entity-Relationship Model
--------------------------------------------------------------------------------

9 Main Entities:

1. Department
   - Attributes: department_id (PK), department_name
   - Stores: ICT, CSE, BBA departments

2. Batch
   - Attributes: batch_id (PK), batch_number, department_id (FK)
   - Relationship: Many-to-One with Department
   - Stores: Batch 1-4 for each department (12 total)

3. Student
   - Attributes: student_id (PK), student_name, department_id (FK), 
     batch_id (FK), email, phone
   - Relationships: Many-to-One with Department and Batch
   - Business Rule: Student ID format (B220102002 → Batch 2)

4. FoodCategory
   - Attributes: category_id (PK), category_name
   - Categories: Breakfast, Lunch/Dinner, Bharta, Snacks/Drinks

5. FoodItem
   - Attributes: food_item_id (PK), food_name, category_id (FK), price, 
     calories, protein, carbs, fat, is_available
   - Relationship: Many-to-One with FoodCategory
   - Contains: 31 real canteen food items with complete nutritional data

6. Invoice
   - Attributes: invoice_id (PK), student_id (FK), created_by, total_price, 
     total_calories, total_protein, total_carbs, total_fat, created_at
   - Relationship: Many-to-One with Student
   - Stores: Purchase transactions with aggregated totals

7. InvoiceDetails
   - Attributes: detail_id (PK), invoice_id (FK), food_item_id (FK), quantity, 
     subtotal_price, subtotal_calories, subtotal_protein, subtotal_carbs, 
     subtotal_fat
   - Relationships: Many-to-One with Invoice and FoodItem
   - Resolves: Many-to-Many relationship between Invoice and FoodItem

8. HealthRecord
   - Attributes: record_id (PK), student_id (FK), height, weight, bmi, 
     recorded_by, notes, recorded_at
   - Relationship: Many-to-One with Student
   - Calculates: BMI = weight / (height/100)²

9. NutritionAnalysis
   - Attributes: analysis_id (PK), student_id (FK), analysis_date, 
     period_start, period_end, avg_daily_calories, avg_daily_protein, 
     avg_daily_carbs, avg_daily_fat, calorie_status, protein_status, 
     advice, analyzed_by
   - Relationship: Many-to-One with Student
   - Stores: Periodic nutrition analysis with recommendations


Normalization
--------------------------------------------------------------------------------

First Normal Form (1NF):
- All tables have atomic values (no multi-valued attributes)
- Each column contains single values
- Primary keys defined for all tables

Second Normal Form (2NF):
- All 1NF requirements met
- No partial dependencies (all non-key attributes depend on entire primary key)
- Example: InvoiceDetails - both invoice_id and food_item_id needed for 
  subtotals

Third Normal Form (3NF):
- All 2NF requirements met
- No transitive dependencies eliminated
- Example: Student table references Department and Batch separately 
  (not department through batch)
- Food nutritional values stored in FoodItem, not duplicated in InvoiceDetails


Constraints & Integrity
--------------------------------------------------------------------------------

Primary Keys: Auto-increment integers or business keys (student_id)

Foreign Keys: All relationships enforced with ON DELETE RESTRICT

CHECK Constraints:
- Price, calories, nutritional values >= 0
- BMI calculated values validate against height/weight
- calorie_status and protein_status ENUM values

Indexes:
- Primary keys (automatic)
- Foreign keys for join performance
- Date fields (created_at, recorded_at) for filtering
- Composite index on (department_id, batch_number)

6. API ENDPOINTS (7 Routes - All Raw SQL)
================================================================================

1. GET /api/students/[studentId]
--------------------------------------------------------------------------------
Purpose: Fetch student details
SQL: SELECT * FROM Student INNER JOIN Department INNER JOIN Batch 
     WHERE student_id = ?
Response: Student profile with department and batch info

2. GET /api/foods
--------------------------------------------------------------------------------
Purpose: Fetch all available food items
SQL: SELECT * FROM FoodItem INNER JOIN FoodCategory WHERE is_available = 1
Response: Foods grouped by category

3. POST /api/invoices
--------------------------------------------------------------------------------
Purpose: Create new invoice with transaction
SQL Operations:
  1. Check/Create Student with batch extraction
  2. Fetch FoodItem details
  3. Calculate subtotals and totals
  4. INSERT INTO Invoice
  5. INSERT INTO InvoiceDetails (multiple)
Transaction: Ensures atomicity (all-or-nothing)

4. GET /api/nutrition/[studentId]?period_days=7
--------------------------------------------------------------------------------
Purpose: Generate nutrition analysis
SQL Queries:
  1. Student info with JOINs
  2. SUM(total_calories), AVG(total_calories), COUNT(*) FROM Invoice 
     WHERE created_at >= DATE_SUB(NOW(), INTERVAL ? DAY)
  3. Latest health record
  4. Recent invoices with food details
Calculations: Daily averages, status determination

5. POST /api/health
--------------------------------------------------------------------------------
Purpose: Add health record
SQL: INSERT INTO HealthRecord with BMI = weight / (height/100)²
Response: BMI category (Underweight/Normal/Overweight/Obese)

6. POST /api/analysis
--------------------------------------------------------------------------------
Purpose: Save nutrition analysis
SQL: INSERT INTO NutritionAnalysis
Data: Period stats, status, recommendations

7. GET /api/reports/departments
--------------------------------------------------------------------------------
Purpose: Department-wise statistics
SQL: Complex JOINs with GROUP BY for BMI and nutrition averages per department


7. KEY SQL QUERIES USED
================================================================================

Complex Queries Demonstrating DBMS Concepts:

1. Daily Calorie Intake (JOIN, GROUP BY, DATE functions)
--------------------------------------------------------------------------------
SELECT s.student_name, DATE(i.created_at) as date, 
       SUM(i.total_calories) as daily_calories
FROM Invoice i 
INNER JOIN Student s ON i.student_id = s.student_id
GROUP BY s.student_id, DATE(i.created_at)
ORDER BY date DESC;

2. Department-wise Average BMI (Multiple JOINs, Aggregation)
--------------------------------------------------------------------------------
SELECT d.department_name, AVG(h.bmi) as avg_bmi, 
       COUNT(DISTINCT s.student_id) as student_count
FROM Student s
INNER JOIN Department d ON s.department_id = d.department_id
LEFT JOIN HealthRecord h ON s.student_id = h.student_id
GROUP BY d.department_id;

3. Nutrition Deficiency Detection (Subquery, HAVING)
--------------------------------------------------------------------------------
SELECT s.student_name, AVG(i.total_protein) as avg_protein
FROM Student s
INNER JOIN Invoice i ON s.student_id = i.student_id
WHERE i.created_at >= DATE_SUB(NOW(), INTERVAL 7 DAY)
GROUP BY s.student_id
HAVING avg_protein < 50;

4. Top Food Items by Revenue (Aggregation, ORDER BY, LIMIT)
--------------------------------------------------------------------------------
SELECT f.food_name, SUM(id.quantity) as total_sold,
       SUM(id.subtotal_price) as revenue
FROM InvoiceDetails id
INNER JOIN FoodItem f ON id.food_item_id = f.food_item_id
GROUP BY f.food_item_id
ORDER BY revenue DESC
LIMIT 10;

8. DATABASE VIEWS (3 Created)
================================================================================

1. vw_student_details
--------------------------------------------------------------------------------
CREATE VIEW vw_student_details AS
SELECT s.*, d.department_name, b.batch_number
FROM Student s
INNER JOIN Department d ON s.department_id = d.department_id
INNER JOIN Batch b ON s.batch_id = b.batch_id;

2. vw_menu_items
--------------------------------------------------------------------------------
CREATE VIEW vw_menu_items AS
SELECT f.*, c.category_name
FROM FoodItem f
INNER JOIN FoodCategory c ON f.category_id = c.category_id
WHERE f.is_available = 1;

3. vw_invoice_summary
--------------------------------------------------------------------------------
CREATE VIEW vw_invoice_summary AS
SELECT i.*, s.student_name, COUNT(id.detail_id) as item_count
FROM Invoice i
INNER JOIN Student s ON i.student_id = s.student_id
LEFT JOIN InvoiceDetails id ON i.invoice_id = id.invoice_id
GROUP BY i.invoice_id;
